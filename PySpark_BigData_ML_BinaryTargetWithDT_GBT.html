2. Feature Extraction and ML Training¶

In this section, I am using PySpark DataFrame functions and ML packages for data preparation, model building and evaluation.

For predicting our binary target, above_threshold, The features I am utilising for this prediction are as follow:

  • SensorCateogory: Categorised Sensor based on the total pedestrian count of each sensor
    • Level - 0,1,2,3,4,5 based on the average pedestrian count of the sensor
  • Hour: To identify which timeframe the hour is in
    • Level - Rushhour, Lunch, Regular
  • Holiday: Whether the date is a public holiday
    • Level - Y/N
  • Weekend: Differentiate the weekend and weekdays
    • Level - 0,1
  • WarmerMonth: Categorise 12 Months into two major categories - December~March in one category, and the rest in another.
    • Level - 0,1

The features that I am not using are as follow:

  • ID: The attribute is only a primary key for identifying the entry
  • Date_Time: The elements of this column are already broken down into different columns.
  • Mdate: The correlation of this feature and the target column is near to 0.
  • Sensor_ID: There are 72 different sensors and it is computationally expensive to have 72 differnent new attributes just for this categorical feature. However, to signify the difference in pedestrian count by different sensors, I added a new column SensorCategory and divided sensor IDs to different levels.
  • Sensor_Name: Similar to Sensor_ID, the attribute is replaced by SensorCategory.
  • Month: Instead of using a categorical feature with 12 different levels, I replaced it using WarmerMonth
  • Day: Instead of using a categorical feature with 7 different levels, I replaced it using Week
  • Time: Replaced it with Hour attribute

1. Confusion Metrics and FPR¶

In [60]:
round = getattr(builtins, 'round')
'''
This function takes the predicted values and original label as the input parameter
and returns confusion metrics for evaluating classification models. 
'''
def computeMetrics(prediction): 
    
    FN = prediction.filter((col('prediction') == 0.0) & (col('label') == 1.0)).count()
    # false negatives 
    FP = prediction.filter((col('prediction') == 1.0) & (col('label') == 0.0)).count()
    # false positives
    TP = prediction.filter((col('prediction') == 1.0) & (col('label') == 1.0)).count()
    # true positives 
    TN = prediction.filter((col('prediction') == 0.0) & (col('label') == 0.0)).count()
    # true negatives 
    
    accuracy = round((TP+TN)/(TP+TN+FP+FN),4)
    # the total number of correct predictions
    precision = round(TP/(TP+FP),4)
    # the proportion of true positive in respect to all the positives we classified
    recall = round(TP/(TP+FN),4)
    # the proportion of true positive in respect to all the positives & negatives we declared
    f1 = round((2*TP)/(2*TP+FP+FN),4)
    # balance of recall and precision
    FPR = round(FP/(FP+TN),4)
    
    return accuracy,precision,recall,f1, FPR

Due to severe class imbalance and the fact that majority of the class is 0 (False), we should evaluate the performance of the model with precision and False Positive Rate.

The precision is the ratio of true positive the model caught out of all the positives it declared. Due to severe class imbalance, the correctness of prediction in this case should be represented by high precision and low False Positive Rate. In this case, both models performed poorly in catching the true positive, yet the amount of false negatives were also very low.

Depends on the purpose of this model, the model can be a two-sided sword: if the client is prone to look for the quiet time for pedestriant, for e.g. construction or traffic facilitation, the model could be an ideal choice. from pyspark.mllib.evaluation import BinaryClassificationMetrics

class CurveMetrics(BinaryClassificationMetrics):
    def __init__(self, *args):
        super(CurveMetrics, self).__init__(*args)

    def _to_list(self, rdd):
        points = []
        # Note this collect could be inefficient for large datasets 
        # considering there may be one probability per datapoint (at most)
        # The Scala version takes a numBins parameter, 
        # but it doesn't seem possible to pass this from Python to Java
        for row in rdd.collect():
            # Results are returned as type scala.Tuple2, 
            # which doesn't appear to have a py4j mapping
            points += [(float(row._1()), float(row._2()))]
        return points

    def get_curve(self, method):
        rdd = getattr(self._java_model, method)().toJavaRDD()
        return self._to_list(rdd)

The ROC curve is a performance measurement for classification problems at various threshold settings. ROC is a probability curve and the area under the curve (AUC) is the degree/measure of separability, showing how much the model is capable of distinguishing between classes - the higher the AUC, the better the model is at accuracy.

The Gradient Boost Tree is performing significantly better at AUC metric in comparison to that of Decision Tree. However, due to severe imbalance of class, where 92% of the entry is 0, the AUC might not be an ideal choice for evaluating the performance of the models.
